## 질문

> 캐시 스탬피드(Cache Stampede)는 왜 생기고, 어떻게 막는가?

## 핵심 답변

캐시 스탬피드는 트래픽이 몰리는 hot key의 캐시가 만료되는 순간, "만료 ↔ 갱신" 사이의 짧은 window에 들어온 다수의 요청이 **각자 독립적으로 "캐시 miss니까 내가 DB를 조회해야 한다"고 판단(check-then-act 레이스)**해서 DB로 한꺼번에 쏟아지는 현상이다. DB는 애플리케이션에 비해 스케일아웃이 어렵기 때문에 이 부하가 병목이 되고, 최악의 경우 DB 장애 → 전체 서비스 장애로 전파된다.

핵심 방어책은 세 축이다. (1) **락/singleflight** — 여러 요청 중 1개만 DB에 보내고 나머지는 대기/재사용, (2) **stale-while-revalidate** — 만료된 옛 값을 삭제하지 않고 응답으로 내주면서 백그라운드로 1개만 갱신, (3) **TTL 지터** — 같은 시각에 올라간 키들의 만료를 흩뿌려 동시 만료 폭탄을 방지. 여기에 스케줄러 기반 사전 갱신과 데이터 성격별 fallback 정책을 더할 수 있다. 이들은 경쟁 관계가 아니라 서로의 빈틈을 메우는 **보완** 관계다.

## 해설

### 1. 왜 N개가 전부 DB를 때리는가 — check-then-act 레이스

window 안에 요청 100개가 들어왔다고 하자. 각 요청은 `캐시 조회 → miss → DB 조회 → 캐시 갱신` 순서로 동작하는데, 이 판단이 **서로 독립적**이다. 아무도 아직 캐시를 채우지 못한 상태에서 100개가 동시에 "캐시가 비었네 → 내가 DB 가야지"라는 같은 결론에 도달한다. 요청들을 서로 조율하는 장치가 없다는 것이 스탬피드의 본질이다.

### 2. 락 / singleflight — "1개만 DB"

100개 중 1개만 락을 잡고 DB를 조회하게 하고, 나머지 99개는 그 결과를 재사용한다. Go의 `singleflight`가 대표적으로, 동일 key에 대한 중복 호출을 하나로 합쳐준다.

- **트레이드오프**: 99개를 블로킹으로 대기시키면 병목이 DB에서 **앱**으로 옮겨갈 뿐이다. 갱신이 조금만 느려져도 스레드/커넥션 풀을 물고 있는 요청이 쌓여 앱 리소스가 고갈된다. 또 락을 잡은 1개가 느리거나 죽으면 99개가 통째로 물린다.

### 3. stale-while-revalidate — "굳이 기다릴 필요가 있나?"

"만료 = 삭제 = 캐시가 비어 있음"이라는 전제를 깨는 것이 핵심이다. 만료돼도 값을 지우지 않으면, 나머지 요청은 **옛날 값을 즉시 응답**으로 내주고 DB도 안 때리고 블로킹도 안 한다. 뒤에서 1개만 백그라운드로 갱신한다. 사용자 체감 지연은 0, DB는 1번만 맞는다.

이걸 구현하려면 "언제 갱신할지"를 알아야 하는데, **값 옆에 논리적 만료 시각(soft TTL) 타임스탬프를 함께 저장**하고, 캐시 자체의 물리적 TTL(hard TTL)은 그보다 길게 둔다. 요청이 들어올 때마다 타임스탬프를 비교한다.

- `now < soft` → 신선함, 그대로 응답
- `soft < now < hard` → 값은 살아있지만 논리적으로 만료 → **옛 값 응답 + 그때 1개만 백그라운드 갱신 트리거**
- `now > hard` → 값이 실제로 사라짐 → 완전 miss → 락으로 폴백

즉 스케줄러가 폴링하는 게 아니라, **요청이 들어온 김에** 판단한다. 값 삭제(hard)와 갱신 신호(soft)를 분리한 것이 트릭이다.

- **구멍**: 응답할 옛 값이 없는 **cold start**(또는 새로 hot해진 키)에서는 무력하다. 이때는 락/singleflight로 "1개만 채우게" 강제하는 것으로 메꾼다. 그래서 락과 stale 기법은 보완 관계다.

### 4. 동시 만료 폭탄 — TTL 지터

하나의 hot key가 아니라, **서로 다른 수천 개의 키가 거의 동시에 만료**되어 쏟아지는 경우도 있다. 배포/캐시 워밍업/트래픽 스파이크 직후 같은 시점에 올라간 키들에 **똑같은 고정 TTL**을 박으면 만료도 같은 초에 몰린다. 해법은 TTL에 무작위성(지터)을 주는 것이다: `TTL = 300s + random(0~60s)`. 단순하지만 실무에서 매우 자주 쓴다.

### 5. 분산 락의 함정

서버가 여러 대인 분산환경에서는 in-process 뮤텍스/singleflight로는 서버 간 조율이 안 된다. 그래서 Redis `SET key val NX EX`로 "내가 갱신 담당"을 하나만 선출하는 분산 락을 쓴다. 여기엔 단계적으로 함정이 있다.

1. **홀더가 죽으면 데드락**: 락 잡은 서버가 DB 조회 중 크래시/단절되면 아무도 락을 풀지 않아 무한 대기. → 락에 **TTL**을 걸어 자동 해제한다(`SET ... NX EX 5`).
2. **작업 시간 > 락 TTL이면 락이 새어나간다**: TTL 5초인데 DB가 느려 갱신이 8초 걸리면, 5초에 락이 풀려 2번 요청이 락을 새로 잡고 또 DB로 간다("1개만" 보장 붕괴, 중복 갱신).
3. **남의 락을 지운다**: 위 상황에서 1번이 8초에 작업을 끝내고 `DEL key`로 반납하면, 그 시점에 꽂혀 있는 락은 이미 **2번의 락**이다. 1번이 2번 락을 지워버려 락이 무주공산이 된다. → 락 값에 **고유 토큰(UUID)**을 넣고, 반납은 "값이 아직 내 UUID일 때만 삭제"를 **Lua 스크립트로 원자적으로** 처리한다.
4. **UUID로도 안 풀리는 것**: UUID는 오삭제만 막지, "작업 > TTL이면 2번이 진입"하는 문제는 그대로다. 이건 별개 문제이고 해법은 (a) **워치독(lease 연장)** — 홀더가 살아있는 동안 주기적으로 TTL 연장(Redisson은 기본 30초 락을 10초마다 갱신), (b) **넉넉한 TTL**, (c) **펜싱 토큰**(단조증가 번호로 낡은 쓰기 거부)이다.

### 6. 한 발 물러난 관점 — 목적이 정합성인가 부하인가

분산 락의 이 토끼굴(Redlock 논쟁 등)은 원래 "돈 이체처럼 절대 2번 실행되면 안 되는" **correctness-critical 락**을 위한 고민이다. 그런데 **캐시 스탬피드 방어의 목적은 correctness가 아니라 부하 감소**다. 어쩌다 락이 새서 2~3개가 DB를 때려도, 수백 개가 1~2개로 줄었으면 이미 목표 달성이다. 따라서 캐시 갱신 락은 "정확히 1개"가 아니라 **"대충 1개(best-effort single-flight)"면 충분**하고, 완벽주의 장치들은 대개 오버엔지니어링이다. 락의 목적이 정합성이냐 부하냐를 구분하는 것이 실무 감각이다.

### 7. 락을 못 잡은 요청은 어떻게 처리할까

락을 못 잡은 요청을 전부 기다리게 하면 DB 부하는 줄어도 애플리케이션의 worker/thread/connection이 묶인다. 반대로 전부 실패시키면 DB는 보호되지만 사용자 경험이 크게 나빠질 수 있다. 그래서 기준은 "응답 가용성이 중요한가, 데이터 최신성/정합성이 중요한가"다.

| 데이터 | 적절한 fallback | 이유 |
|---|---|---|
| 상품 상세 조회수, 랭킹, 추천 목록 | stale 반환 | 조금 오래된 값을 보여줘도 피해가 작고, 지연을 줄이는 것이 더 중요하다. |
| 상품 가격, 재고 노출 | 짧게 대기 후 실패/대체 처리 | 틀린 값을 보여주는 비즈니스 리스크가 있으므로 무조건 stale을 주기 어렵다. |
| 사용자 권한, 관리자 작업 허가 | fail-closed | 불확실할 때 허용하면 보안 사고가 되므로 기본은 차단이다. |

특히 권한 판단에서는 **fail-open**과 **fail-closed**를 구분해야 한다. fail-open은 판단 시스템이 불안정해도 요청을 허용하는 정책이고, fail-closed는 불안정하면 차단하는 정책이다. 민감 리소스나 관리자 작업은 보통 fail-closed가 맞지만, 공개 리소스까지 무조건 fail-closed로 막으면 권한 시스템 장애가 서비스 전체 가용성 장애로 번질 수 있다. 즉 "권한 데이터"도 하나의 전역 정책이 아니라 공개 리소스, 로그인 사용자 리소스, 민감 리소스, 관리자 작업처럼 리스크별로 나누어야 한다.

## 예시

```text
# stale-while-revalidate 판단 로직 (의사코드)
value, soft_expire = cache.get(key)          # 값 + 논리적 만료 시각을 함께 저장
if value is None:                            # hard TTL도 지남 → 완전 miss
    if lock.try_acquire(key):                # 1개만 DB로
        value = db.query(key)
        cache.set(key, value, soft=now+300, hard_ttl=360)
        lock.release(key)
    else:
        value = wait_or_serve_fallback()
elif now > soft_expire:                       # 값은 있으나 논리적으로 만료
    if lock.try_acquire(key):                 # 1개만 백그라운드 갱신
        async_refresh(key)
    return value                              # 나머지는 옛 값 즉시 응답
else:
    return value                              # 신선 → 그대로
```

```text
# Redis 분산 락 반납 — 원자적 check-and-delete (Lua)
if redis.call("GET", KEYS[1]) == ARGV[1] then   -- ARGV[1] = 내 UUID
    return redis.call("DEL", KEYS[1])
else
    return 0                                     -- 남의 락이면 건드리지 않음
end
```

## 꼬리 질문

- 락 방식에서 대기하는 99개를 블로킹시키면 앱 리소스가 고갈되는데, 어떻게 설계해야 하나? (재시도 백오프 / stale 폴백)
- soft TTL / hard TTL을 각각 얼마로 잡아야 하나? 데이터 신선도 요구와 어떻게 절충하나?
- 분산 락에서 "작업 시간 > 락 TTL"을 워치독으로 푸는 것과 넉넉한 TTL로 푸는 것의 트레이드오프는?
- 락을 못 잡은 요청은 언제 stale을 반환하고, 언제 대기하거나 실패해야 하는가?
- 권한 확인처럼 보안 판단이 들어간 캐시는 fail-open과 fail-closed 중 무엇을 기본으로 해야 하는가?
- 펜싱 토큰이 정말 필요한 상황은 언제이고, 캐시 갱신에는 왜 대체로 과한가?
- 스케줄러 기반 사전 갱신(만료 전 미리 채우기)은 언제 stale-while-revalidate보다 유리한가?

## 실무 연결

- **장애 대응**: hot key(인기 상품, 실시간 랭킹, 메인 배너)의 캐시 만료가 DB 커넥션 풀 고갈 → DB 부하 급증 → 전체 장애로 전파되는 시나리오는 실제 대형 장애의 흔한 패턴이다. 스탬피드 방어는 DB를 보호하는 방파제다.
- **성능/비용**: stale-while-revalidate는 p99 지연을 크게 낮춘다(갱신 대기 없이 옛 값 즉시 응답). 약간의 데이터 지연을 허용할 수 있는 도메인(추천, 목록, 통계)에서 특히 유효하다.
- **도메인 정책**: 같은 캐시 장애라도 조회수/추천 목록은 stale로 버틸 수 있고, 가격/재고는 짧은 대기나 실패가 나을 수 있으며, 권한/관리자 작업은 보통 fail-closed가 맞다. 캐시 전략은 기술 문제가 아니라 도메인 리스크 판단까지 포함한다.
- **시스템 설계**: 분산 락을 도입할 때는 "이 락의 목적이 정합성인가 부하 감소인가"를 먼저 판단해야 한다. 캐시 갱신은 후자이므로 best-effort로 단순하게 가고, 결제·재고 차감 같은 정합성 락에만 워치독/펜싱 등 무거운 장치를 쓴다.
